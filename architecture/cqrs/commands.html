Commands ​

¿Cuándo utilizar un command? ​

Como se comentó en el artículo de Introducción sobre CQRS los commands deben utilizarse siempre que estemos hablando de operaciones de tipo PUT, DELETE, PATCH y POST.

¿Cómo creo y manejo un command? ​

Todos los commands deberán estar ubicados en el proyecto de Application. Tomando como caso de uso, la creación de un usuario, la estructura debería quedar de esta manera:

Ubicación ​

plaintext
📂 Application
│   ├── 📂 Users
│   │   ├── 📂 Commands
│   │   │   ├── 📂 Create
│   │   │   │   └── 📄 CreateUserCommand.cs
|   |   |   |   └── 📄 CreateUserCommandHandler.cs
|   |   |   |   └── 📄 CreateUserCommandValidator.cs

Command ​

Dentro de nuestro CreateUserCommand irá todo el contenido que solicitaremos desde nuestro endpoint: nombre, correo electrónico, número de teléfono, entre otros.

INFO

Por tema de fácil inacialización y manejo del objeto, es recomendable que nuestros Command sean de tipo public record.

csharp
public record CreateUserCommand(
    string FullName,
    string Email,
    string PhoneNumber,
    string JobTitle,
    string ProfilePicture,
    DateTime DateOfBirth
) : ITransactionalCommand<long>;

Es importante notar que para que ingrese dentro del Request Pipeline, nuestro command debe heredar de una clase de tipo IRequest. En este caso ITransactionalCommand no es más que una re-interpretación de esta clase, la cual es utilizada para identificarla y ser manejada por un Pipeline Behavior.

csharp
internal interface ITransactionalCommand<TResponse> 
    : IBaseTransactionCommand, IRequest<Result<TResponse>>;

Handler ​

El handler es quien se encargará de manejar la lógica de nuestro command, ya sea consultas a base de datos, manejo de errores y quien devuelva un Result a nuestra capa de Presentation.

csharp
public sealed class CreateUserCommandHandler(
    IUnitOfWork unitOfWork,
    IUserRepository userRepository
) : IRequestHandler<CreateUserCommand, Result<long>>
{
    public async Task<Result<long>> Handle(
        CreateUserCommand command,
        CancellationToken cancellationToken)
    {
        var emailExists = await userRepository.EmailExistsAsync(command.Email, cancellationToken);
        
        if (emailExists)
        {
            return Result.Failure<long>(UserErrors.EmailNotUnique, HttpStatusCode.Conflict);
        }
        
        var user = User.Create(
            command.FullName,
            command.Email,
            command.PhoneNumber,
            command.JobTitle,
            command.ProfilePicture
        );
        
        await userRepository.InsertAsync(user, cancellationToken);
        
        await unitOfWork.SaveChangesAsync(cancellationToken);
        
        return Result.Success(user.Id);
    }
}

Validator ​

Haciendo uso de la librería FluentValidation, nuestro validator tendrá todas las validaciones que necesitan cumplirse previo a ingresar dentro del Handler. Si obtuviésemos mínimamente un error de validación, nuestro ValidationPipelineBehavior va a devolver un status 400 (BadRequest).

csharp
internal sealed class CreateUserCommandValidator 
    : AbstractValidator<CreateUserCommand>
{
    public CreateUserCommandValidator()
    {
        RuleFor(x => x.FullName)
            .NotEmpty()
            .MaximumLength(125)
            .WithErrorCode(UserErrorCodes.Common.FullNameMaxLengthExceeded);
        
        RuleFor(x => x.Email)
            .NotEmpty().WithErrorCode(UserErrorCodes.Common.MissingEmail)
            .EmailAddress()
            .WithErrorCode(UserErrorCodes.Common.InvalidEmail);
        
        RuleFor(x => x.DateOfBirth)
            .Must(x => x <= DateTime.UtcNow.AddYears(-18))
            .WithErrorCode(UserErrorCodes.Create.MinimumAgeRequired);
    }
}
